Create a Resource Controller

在使用 Spring 来创建 RESTful Web 服务的方式中， HTTP 请求由 controller 处理。 这些组件由 @RestController 标识：

  • @GetMapping 注解确保对 /greeting 的 HTTP Get请求会被映射到 greeting() 方法。

  • @RequestParam 注解将查询字符串中的 name 参数绑定到 greeting() 方法的name参数中。

  • @RestController 注解标记了该类是一个返回域对象而不是视图的 controller。是 @Controller 和 @RequestBody 的简写。

  • Greeting对象必须转换成 JSON 才能响应回客户端。不过由于有 Spring 的 HTTP 消息转换器的支持，所以无需手动配置。因为 Jackson2 被依赖添加到 classpath 中，所以 Spring 会自动选择使用MappingJackson2HttpMessageConverter将Greeting实例转换为 JSON。

  • @SpringBootApplication 是一个方便的注解，它包括了一下注解:

    • @Configuration: 将类标记为程序上下文 bean 的定义源。
    • @EnableAutoConfiguration: 告诉 SpringBoot 根据 classpath 设置 、其他 bean 和 各种属性配置开始添加 bean。例如，如果 spring-webmvc在 classpath 里，该注解就会将应用标记为 web 程序，并激活一些关键行为例如设置DispatcherServlet。
    • @ComponentScan: 告诉 Spring 在当前包内查找其他 component 、 configuration 、 service 、 controller。

Sharding-JDBC 介绍

来自官网：Sharding-JDBC

定位为轻量级Java框架，在Java的JDBC层提供的额外服务。 它使用客户端直连数据库，以jar包形式提供服务，无需额外部署和依赖，可理解为增强版的JDBC驱动，完全兼容JDBC和各种ORM框架。

  • 适用于任何基于JDBC的ORM框架，如：JPA, Hibernate, Mybatis, Spring JDBC Template或直接使用JDBC。
  • 支持任何第三方的数据库连接池，如：DBCP, C3P0, BoneCP, Druid, HikariCP等。
  • 支持任意实现JDBC规范的数据库。目前支持MySQL，Oracle，SQLServer，PostgreSQL以及任何遵循SQL92标准的数据库。

Redis缓存问题(击穿、穿透、雪崩)及解决方案

Posted on 2020-09-18 Edited on 2020-11-01 Redis 缓存问题

缓存雪崩

介绍

大量缓存数据集体失效，导致大量请求直接打到数据库，造成数据库压力过大导致响应不及时，甚至会使数据库宕机进而整个系统的崩溃。

解决方案

  1. 在设置key的时候，随机生成失效时间
  2. 在基于集群的场景下，可以将热点key平均分散到不同节点上，可以分散雪崩的压力
  3. 不设置失效时间(永久生效)
  4. 定时任务刷新key，在key即将过期时刷新失效时间

缓存穿透

缓存穿透和缓存击穿这两个概念很容易混淆。

介绍

个人认为缓存穿透的重点在于透，目的就是透过Redis直接访问数据库，是一种恶意攻击行为，方法是获取明显不存在的数据，如：自增主键(0开始)下请求获取id小于0的数据。

解决方案

  1. 将空数据放到Redis里，只针对某一个参数
  2. 拉黑请求IP，不影响恶意用户切换IP
  3. 参数过滤，校验参数合法性
  4. 布隆过滤器

缓存击穿

介绍

某一个热点数据失效，导致大量请求直接访问数据库(与穿透不同的是，此处请求的资源是存在的)。

解决方案

  1. 热点key永不过期
  2. 分布式锁(分布式: ZoKeeper或Redis)/互斥锁(单体)
    • 保证只有一个请求去访问数据库
    • 该请求将数据添加到缓存中
    • 其他请求从缓存中获取并返回

Spring Cloud简介

Spring Cloud为开发者提供了快速构建分布式系统的常见模式(如配置管理，服务发现，断路器，智能路由，微代理，控制总线，一次性令牌，全局锁，领导选举，分布式会话，群集状态)的工具。
官网原文：
Spring Cloud provides tools for developers to quickly build some of the common patterns in distributed systems (e.g. configuration management, service discovery, circuit breakers, intelligent routing, micro-proxy, control bus, one-time tokens, global locks, leadership election, distributed sessions, cluster state).

特征

Spring Cloud专注于为典型的用例和可扩展性机制（包括其他用例）提供良好的开箱即用体验。

  • Distributed/versioned configuration 分布式/版本化的配置管理
  • Service registration and discovery 服务注册与服务发现
  • Routing 路由
  • Service-to-service calls 端到端的调用
  • Load balancing 负载均衡
  • Circuit Breakers 断路器
  • Global locks 全局锁
  • Leadership election and cluster state 选举与集群状态管理
  • Distributed messaging 分布式消息

Java-线程池

Posted on 2020-08-22 Edited on 2020-11-07 Java

API接口及实现类

类型 名称 描述
interface Executor 最上层的接口，定义了执行任务的方法execute(Runnable)
interface ExecutorService 扩展Executor接口，添加了Callable支持，定义了任务提交方法submit及关闭方法shutdown
interface ScheduledExecutorService 扩展了ExecutorService接口，增加定时任务相关方法
class ThreadPoolExecutor 标准的线程池实现类
class ScheduledThreadPoolExecutor 继承ThreadPoolExecutor，实现ScheduledExecutorService，实现定时任务方法

定时任务接口中定义的两个周期执行方法区别：

  • scheduleAtFixedRate方法：在给定延迟后开始执行，如果遇到异常，则禁止后续执行。若执行时间大于执行周期，则当前任务执行完成后，马上执行下次任务。
  • scheduleWithFixedDelay：给定延迟后开始执行，每次任务执行完成后，经过执行周期时间，再执行下次任务。

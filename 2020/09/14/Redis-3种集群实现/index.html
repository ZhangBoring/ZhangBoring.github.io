Redis-3种集群实现

Posted on 2020-09-14 Edited on 2020-11-07 Redis

Redis三种集群介绍

Redis由三种集群方式：主从(Master-Replica)复制、哨兵(Sentinel)模式和Cluster集群。
配置文件及SpringBoot项目源码：源码地址

主从复制

主从复制，是指将一台Redis服务器的数据(主Master)单向复制到另一台Redis服务器(从Replica)。

作用

  1. 数据冗余：实现了数据的热备份，是持久化之外的一种数据冗余方式。
  2. 故障恢复：当主节点出现故障时，可以由从节点提供服务，实现快速的故障恢复，服务冗余增加高可用性。
  3. 负载均衡：主从复制+读写分离，由主节点负责写，从节点负责读，实现负载均衡。写应用连接主节点，读应用连接从节点。
  4. 高可用：主从复制是哨兵和集群的基础。

启用

主从复制有三种启动方式：

  • 在配置文件中配置 replicaof host port
  • 在Redis-Server启动是加启动参数 --replicaof host port
  • 在Redis-Cli中执行命令 replicaof host port

过程

主从复制过程大致分为3个步骤:

  1. 连接建立阶段
    保存主节点信息，建立socket连接，并进行权限验证。
  2. 数据同步阶段
    连接建立完成后，从节点发送psync命令，主节点项从节点复制数据。
  3. 命令传播阶段
    主节点将自己执行的写命令发送给从节点。

哨兵模式

介绍

Redis的哨兵(Sentinel)系统用于管理多个Redis服务器，该系统执行以下三个任务：

  1. 监控(Monitoring)：Sentinel会不断检所有主从服务器是否正常。
  2. 提醒(Notification)：当被监控的某个Redis服务器出现故障，Sentinel可通过API向管理员或其他应用程序发送通知。
  3. 自动故障迁移(Automatic faileover)：当一个主服务器不能正常工作时，Sentinel会开始一次故障迁移操作，它会将失效的主服务器的从节点中的一个升级为新的主节点，并让失效的主节点的其他从节点改为复制新的主服务器。当客户端试图连接失效的主节点时，集群也回想客户端返回新主服务器的地址。
    • 仅当哨兵1发现主节点故障时，系统并不会马上faileover，这个现象称为主观下线。只有当一定数量的哨兵发现检测到主服务器故障时，哨兵之间会发起一次投票，选举新的主节点，结果由一个哨兵发出，进行failover操作，称为客观下线。

Cluster(集群)

介绍

Redis集群是一个提供在多个Redis节点间共享数据的程序集。

优缺点

  1. Redis集群不支持处理多个key的命令，因为多key的命令需要在多个节点间移动数据，导致性能下降，且在高并发场景下会出现不可预料的错误。
  2. Redis集群通过分区实现一定程度的可用性，可以在某个节点宕机的情况下继续处理命令。优势：
    • 自动分割数据到不同的节点上
    • 集群的部分节点不可达情况下可继续处理命令

数据分片

Redis集群没有使用一致性hash，而是引入了哈希槽的概念。
Redis集群中有16,384(2^14)个哈希槽，每个key通过CRC16校验后对16,384取模决定防止在哪个槽内，集群的每个节点负责一部分槽。
例如，当前集群中有3个节点：

  • A节点负责0-5500号槽
  • B节点负责5501-10000号槽
  • C节点负责10001-16383号槽

这种结构很容易添加/删除节点，如果想添加D节点，则从A、B、C节点中分出部分槽到D上；想删除C节点，则将C节点上的槽分到A、B节点上，然后将没有数据的C节点移除即可。移动哈希槽的操作不会导致服务不可用。
